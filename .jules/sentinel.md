# Sentinel's Journal: Critical Security Learnings

## 2026-07-10 - Parameter Injection via Upstream Proxy URL Parsing
**Vulnerability:** The `parseGithubRepoUrl` function was susceptible to parameter injection because it extracted the owner and repository name segments from raw URLs without stripping query parameters (`?`) or hash fragments (`#`) first. Downstream, the server-side API `/api/github-meta` concatenated these segments directly into a fetch request to `https://api.github.com/repos/${owner}/${repo}`. An attacker could pass a crafted GitHub URL (e.g., `https://github.com/owner/repo?some-query`) to manipulate the upstream GitHub API request parameters.
**Learning:** Naive regex matching that is not explicitly anchored or fails to strip URL components (like query strings and fragments) can leak unvalidated segments into downstream HTTP requests. This leads to parameter injection where client input modifies the parameters, query, or path of an upstream request initiated by the server.
**Prevention:**
1. Always split or strip URL queries (`?`) and hash fragments (`#`) before performing regex parsing on URL paths.
2. Apply strict character whitelists and length limits to matched segments before trusting them in downstream API requests (e.g., restricting GitHub owners to `/^[a-zA-Z0-9-]+$/` and max 39 characters, and repos to `/^[a-zA-Z0-9-_.]+$/` and max 100 characters).

## 2026-07-10 - Proxy Endpoint Rate Limit Exhaustion and Denial of Service
**Vulnerability:** Unauthenticated server-side proxy endpoints (such as `/api/github-meta`) that relay requests to third-party APIs (like GitHub) without client-side rate limiting are susceptible to quota/token exhaustion and Abuse/Denial of Service. Attackers can spam these routes to drain pool limits or API keys, disabling core application features for all other users.
**Learning:** Proxy endpoints designed to keep third-party APIs out of browser CSP must have independent, server-side ingress control (such as IP-based rate limiting) to prevent cascading resource starvation or rate-limiting on upstream APIs.
**Prevention:** Always bound unauthenticated proxy endpoints to tight IP-based sliding-window rate limits using local database-backed atomic operations (e.g. `checkRateLimit`) to prevent bad actors from abusing server egress budgets.

## 2026-07-10 - CPU-Bound Pruning Logic in In-Memory Rate Limiting (Self-Inflicted DoS)
**Vulnerability:** In-memory sliding-window rate limiters can introduce severe CPU bottlenecks if the map of keys is pruned manually via full iteration `O(N)` on every request when the size exceeds a certain threshold. Under high load, this O(N) traversal blocks the Node.js single-threaded event loop, leading to a self-inflicted Denial of Service (DoS) even under normal or moderate traffic volumes.
**Learning:** Manual cache eviction and timing-based filtering are inefficient and prone to timing/CPU exhaustion. Additionally, in serverless environments (like Vercel), in-memory states are isolated and not shared across serverless instances, yielding weaker protection.
**Prevention:** Always bound unauthenticated public or proxy endpoints to tight IP-based sliding-window rate limits using local database-backed atomic operations (such as the project's existing `checkRateLimit` helper) to prevent bad actors from bypassing limits across serverless instances.

## 2026-07-10 - Schema Validation Bypass on Dual-Protocol Endpoints
**Vulnerability:** The write operations inside the JSON-RPC Model Context Protocol (MCP) server endpoint (`/api/mcp`) were bypassing the strict Zod validation schemas established for the REST API endpoints (`/api/skills`, `/api/vibes`, `/api/blog`, and forum replies). This allowed clients connecting via MCP write tools to submit payloads with unvalidated, excessively long, or malformed inputs (e.g. extremely long strings, prompts arrays, or unvalidated imageUrls), circumventing the application's ingress constraints and risking database bloat or Denial of Service (DoS).
**Learning:** When exposing parallel protocol handlers (e.g., REST and JSON-RPC/MCP) on top of the same data mutation layers, each protocol entry point must consistently enforce the same schema validations. Relying only on REST route-level schema validations creates parallel access vectors that bypass those safety policies.
**Prevention:** Always centralize or share Zod validation schemas across all API endpoints (REST and RPC) and validate all incoming write parameters at the protocol boundary (e.g., using `safeParse` inside the tool call dispatcher) before forwarding them to the database layer.

## 2026-07-11 - PostgreSQL ILIKE Escape Bypass and Pattern Injection
**Vulnerability:** The SQL search sanitization utility `sanitizeSearchTerm` failed to strip backslashes (`\\`). Because PostgreSQL uses the backslash as the default escape character in `LIKE`/`ILIKE` search patterns, an attacker could supply unescaped backslashes to escape wildcards or manipulate the SQL matching structure, potentially leading to search pattern manipulation or database-level syntax errors.
**Learning:** Preventing wildcard bypass and injection in SQL `LIKE`/`ILIKE` requires stripping or escaping the escape character itself (defaulting to `\\` in PostgreSQL). Simply stripping SQL wildcards (`%` and `_`) is insufficient if the escape character is left intact to override normal syntax.
**Prevention:** Always include backslashes (`\\\\`) in character whitelists/blacklists when sanitizing search inputs targeted for PostgreSQL pattern matching queries.

## 2026-07-12 - Denial of Service (DoS) and Database Bloat via Unconstrained Nested Parameters
**Vulnerability:** Although core write endpoints validated request payloads using Zod schemas, nested parameters such as `tags` arrays, `tools` arrays, `prompts` arrays, and the `systemPrompt` text parameter lacked inner string element length boundaries or array limit constraints. This would allow an attacker to send extremely large nested values (e.g. megabytes of text as a single tag or thousands of items in the prompts array) to cause severe database bloat and memory exhaustion/Denial of Service (DoS) when retrieving and parsing these records.
**Learning:** Checking a field's presence or validating it merely as a generic string/array is insufficient. Validation boundaries must enforce deep defenses: every collection or nested list element must be explicitly bounded in both count (maximum array size) and character size (maximum string length) to prevent memory or persistence exhaustion attacks.
**Prevention:** Always restrict individual list elements to a reasonable length (e.g., maximum of 50 characters for tags/tools) and impose hard ceilings on arrays of nested parameters (e.g., maximum of 20 elements for prompts) to guarantee deterministic memory usage.

## 2026-07-14 - Denial of Service (DoS/ReDoS) via Unconstrained Input String Lengths
**Vulnerability:** Public, unauthenticated endpoints (like the database-backed catalog search or the server-side GitHub URL parsing proxy `/api/github-meta`) accepted strings of arbitrary lengths. Processing excessively large raw strings (e.g., 50,000+ characters) on these routes created a severe risk of CPU and memory exhaustion. Specifically, executing regex parsing with complex optional patterns on unbounded URLs can trigger exponential backtracking (ReDoS), while passing unbounded search queries to downstream database engines blocks thread loops and causes database thread bloat.
**Learning:** Sanitizing inputs for characters (such as SQL metacharacters or regex injection tokens) is insufficient if the string's length is not strictly constrained at the earliest validation boundary. Public, unauthenticated routes must enforce strict, tight length bounds matching legitimate usage thresholds before performing any CPU-bound string manipulation, regex matches, or query compilation.
**Prevention:** Always truncate search query variables to a practical limit (e.g., 100 characters) before running regex replacements/matching, and enforce absolute maximum string lengths (e.g., 500 characters) on any user-supplied URLs prior to running parsing logic or regex evaluations.

## 2026-07-16 - Image Host Validation Bypass in Dual-Protocol schemas (SSRF and DoS Vector)
**Vulnerability:** The schema `blogPostSchema` validated that `imageUrl` was a valid URL but did not enforce validation against the allowed hostnames array in `allowedImageHosts.ts`. Since blog posts can be submitted by both bearer-token agents and users across standard REST endpoints and the MCP server, an attacker could submit a blog post with an image URL pointing to an arbitrary domain.
**Learning:** Next.js `next/image` throws a fatal rendering exception if it attempts to load an image from an unconfigured host. Accepting arbitrary unvalidated image URLs on public/agent mutation vectors can therefore trigger a client-side Denial of Service (DoS) for all visitors viewing the resource. Additionally, this introduces third-party pixel tracking and SSRF exposure.
**Prevention:** Always restrict submitted user/bot image URLs to the unified list of allowed image hostnames via `.refine(isAllowedImageUrl)` across all schemas (including `blogPostSchema`), maintaining parity with `next.config.ts`'s `remotePatterns` and CSP settings.
